# 04. DB 設計

Supabase 用の PostgreSQL マイグレーション SQL をそのまま貼れる粒度で記述する。
全テーブルに RLS を設定。インデックスも明示。

`supabase/migrations/YYYYMMDDHHMMSS_create_xxx.sql` の形でファイル分割推奨。

## 0. 共通: トリガー関数

```sql
-- updated_at 自動更新（プロジェクトに 1 つ）
CREATE OR REPLACE FUNCTION public.update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

## 1. ユーザー / プロフィール（最小）

```sql
-- profiles
CREATE TABLE public.profiles (
  user_id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  display_name TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO public.profiles (user_id, display_name)
  VALUES (NEW.id, COALESCE(NEW.raw_user_meta_data->>'name', split_part(NEW.email, '@', 1)));
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();

ALTER TABLE public.profiles ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Read all profiles" ON public.profiles FOR SELECT TO authenticated USING (true);
CREATE POLICY "Update own profile" ON public.profiles FOR UPDATE TO authenticated USING (auth.uid() = user_id);
```

## 4. 選択肢の木

### 4.1 ノード（木構造）

```sql
CREATE TABLE public.selection_tree_nodes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  parent_id UUID REFERENCES public.selection_tree_nodes(id) ON DELETE CASCADE,
  category public.quote_category NOT NULL,    -- どの見積カテゴリの木か
  label TEXT NOT NULL,                         -- 表示ラベル
  sort_order INT NOT NULL DEFAULT 0,

  -- 役割 A: ナビゲーション + 値の保有。ノードマスタ値（属性付き）への参照
  node_master_value_id UUID REFERENCES public.node_master_values(id) ON DELETE SET NULL,

  -- 役割 D: 計算式テンプレート（葉の場合のみ）
  formula_template_id UUID REFERENCES public.formula_templates(id) ON DELETE SET NULL,

  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_selection_tree_nodes_parent ON public.selection_tree_nodes(parent_id);
CREATE INDEX idx_selection_tree_nodes_category ON public.selection_tree_nodes(category);
CREATE INDEX idx_selection_tree_nodes_formula ON public.selection_tree_nodes(formula_template_id);
CREATE TRIGGER update_selection_tree_nodes_updated_at BEFORE UPDATE ON public.selection_tree_nodes
  FOR EACH ROW EXECUTE FUNCTION public.update_updated_at_column();

ALTER TABLE public.selection_tree_nodes ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Read all" ON public.selection_tree_nodes FOR SELECT TO authenticated USING (true);
CREATE POLICY "Write" ON public.selection_tree_nodes FOR ALL TO authenticated USING (true) WITH CHECK (true);
```

**判定ルール（葉 vs 中間）**:
- `formula_template_id IS NOT NULL` → 葉
- `formula_template_id IS NULL` → 中間（子を持つ）

**マスタ参照と固定変数の扱い**:
旧設計の役割 B（master_entity / master_id polymorphic FK）と役割 C（fixed_variables JSONB）は **廃止**。マスタ参照値も固定変数も、すべて `node_master_values.values JSONB` の属性として持たせる方式に統一した。これにより:

- 「材料マスタ」「工程マスタ」のような独立テーブルは不要（→ §6 で削除）
- ノードに紐づく属性は `attribute_schema` で型付きに定義可能
- 計算式は `path_key` を介して属性を取り出すだけで動く

## 6. 材料・工程の扱い（独立テーブルなし）

材料マスタ・工程マスタといった独立テーブルは持たない。代わりに **ノードマスタ値の属性** として持たせる:

- 「材料種別」ノードマスタ → `{title, category, density, unit_price}`
- 「工程手法」ノードマスタ → `{title, hourly_rate, setup_time}`
- 「表面処理手法」ノードマスタ → `{title, unit_cost}`
- 「熱処理手法」ノードマスタ → `{title, fixed_cost}`
- 「購入品種別」ノードマスタ → `{title, unit_price}`

これらの属性は計算式から PATH 変数として参照される（`variable_definitions.path_key` で属性キーを指定）。

→ ノードマスタ一本化により、マスタの種類追加・属性追加は `attribute_schema` の編集だけで完結する。

## 11. 注意事項

- **RLS は全テーブルで有効**。デモは「authenticated なら全 R/W」の超ゆるい設定。本番では絞る
- **`items.created_by` / `quotes.created_by`** は `auth.users` への soft reference。FK 制約は付けない（認証状態に依存しない）
- **ENUM 型** は値の追加に CREATE OR ALTER が必要（PostgreSQL 流）
- **JSONB のスキーマ** はアプリケーション側で TypeScript 型で管理する（zod スキーマと突き合わせ）
- 数値型は `NUMERIC` 統一（精度確保）。`NUMERIC(12, 2)` 等の桁制限はデモでは不要
